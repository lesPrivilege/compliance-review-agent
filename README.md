# Compliance Review Agent

合规审查 Agent：知识密集型合规推理的最小可审计实现。

## 这是什么

演示项目：用 Agent 技术处理**知识密集型**的合规审查任务。与 `contract-approval-agent`（路由+HITL）互补。

**核心论点**：合规审查 Agent：知识密集型合规推理的最小可审计实现。固定检索管线保证最低召回，LLM 在 ReAct 循环中按需补充检索（最多两轮）——先查法规、判断证据是否充分、不足再搜。LLM 识别的风险点与确定性规则互补叠加，得出低/中/高三级风险；中高风险经 interrupt 转人工确认。

**检索口径**：这里不是否定向量库，而是把向量、BM25、关键词和 metadata filter 都降级为 agent 工具箱里的检索工具，由 agent 按需调用和复查。向量库解决语料没结构的问题；分类体系、metadata 抽取和文档标准化是在源头上给语料造结构，两者互补而非替代。

**诚实口径**：场景设计参考多行业合规审查的通用模式，数据全部虚构。

> **配套项目**：本项目是三个项目之一——`contract-approval-agent`（合同审批路由，规则+HITL）是结构化决策的互补对照，`agent-quality-workbench`（PM 侧评估工具）提供立项评分和跨 agent 质量仪表盘。

> **在线预览**：[GitHub Pages](https://lesprivilege.github.io/compliance-review-agent/)（verification log）

## 与 contract-approval-agent 的对比

| 维度 | contract-approval-agent | compliance-review-agent |
|------|------------------------|------------------------|
| 核心能力 | 路由 + HITL | RAG + 推理 + 评估 |
| 决策类型 | 「走哪条路」（结构化） | 「是否合规」（知识密集） |
| 工具使用 | 查表（审批矩阵） | 检索（法规/政策/案例） |
| 输出 | 审批路径 + 决策 | 合规报告 + 风险评分 |
| 推理模式 | 规则优先 + LLM 兜底 | LLM 多步推理 + 规则校验 |

### 每个节点的实现方式

| 节点 | 方式 | 为什么 |
|------|------|--------|
| **Ingest** | 解析输入材料，提取关键条款 | 结构化为可检索的片段 |
| **Retrieve** | 法规检索工具（关键词匹配实现） | 合规判断需要参考外部知识，检索是 agent 可按需调用的工具 |
| **Analyze** | LLM ReAct 循环（有 key）/ 确定性回退（无 key） | 真实多步推理：LLM 决定是否需要额外检索，最多 2 轮 |
| **Score** | 确定性规则 + LLM 风险因素叠加 | 业务规则和 LLM 判断互补，不是替代 |
| **HITL** | `interrupt()` + `Command(resume=...)` | 中高风险审查必须人工确认 |
| **Report Gen** | 结构化报告生成 | 输出需要可读、可审计 |

## 数据设计

所有数据虚构，覆盖四板块的通用合规场景。

### 法规库 (`data/regulations/`)

法规是最适合预建、最不适合每次外搜的语料：权威源少而固定，变更低频，自带条文编号作为天然引用锚点。合规结论需要复现「当时依据的具体条文版本」，所以本项目使用本地法规库而不是运行时 web search；真实生产中应补充官方源监控、版本化入库、生效/废止日期、效力层级和查询时点过滤。

| 文件 | 覆盖板块 | 审查范围 |
|------|----------|----------|
| 反垄断合规.md | 通用 | 关联交易、市场支配地位、数据隔离 |
| 财务合规审查.md | 通用 | 预算、付款、发票、税务 |
| 数据隐私合规.md | D（借贷担保） | 数据处理、跨境传输 + 借贷担保合规 |
| 关联交易审查.md | 通用 | 关联方定价、审批程序 |
| 特许经营权合规.md | A（受监管） | 招投标合规、经营权使用费、交叉补贴 |
| 投资评审与反垄断申报.md | B（投资驱动） | 投资评审落地、反垄断申报阈值、供应商准入 |
| 反垄断价格合规.md | C（竞争型） | 价格歧视、掠夺性定价、消费者权益 |

### LLM ReAct 分析机制

`analyze_node` 在有 `OPENAI_API_KEY` 时启用真实 LLM ReAct 循环：
1. 将 `retrieve_node` 已检索的法规 + 材料信息交给 LLM（`bind_tools`）
2. LLM 判断现有证据是否足够；不够则调用 `search_regulations_tool` 再检索
3. 最多 2 轮（硬编码上限，防死循环）
4. 循环结束后 `with_structured_output(ComplianceAnalysis)` 强制输出结构化结论

这个设计刻意强调 **按需搜索与证据闭环**：固定 top-k 只能把一次检索结果交给模型，本项目让模型在证据不足时主动换查询词、补检索，并把引用法规写回结构化输出。

### 检索：两层设计

检索是两层设计而非一层：`retrieve_node` 的固定管线负责保底召回，`analyze_node` 的 ReAct 循环让模型按需调用 `search_regulations` 工具。M012 验证中，模型首轮检索零结果后自行改写查询词再搜——按需检索的价值不在替代固定管线，而在处理「查一次不够」的场景。

当前检索为关键词匹配：法规库仅 7 篇、结构规整，关键词已足够；向量检索留待语料规模真正需要时引入，而不是默认引入。

### 置信度语义迭代

置信度语义经过一次真实迭代：初版 prompt 下，法规库零结果时模型仍给出 0.82 的置信度——它把「对结论的信心」与「法规依据是否充分」混为一谈。将 prompt 语义改为后者并重验，降至 0.15。完整过程见 `docs/llm-verification-log.md`。

`score_node` 将 LLM 得出的 `risk_factors` **并入**确定性规则的风险因素列表——LLM 判断和业务规则判断互补叠加，不是替代。

无 API key 时自动回退到原有确定性逻辑，所有测试不受影响。LLM 测试需 `OPENAI_API_KEY`，无 key 时自动跳过。

## 设计决策

### 为什么是合规审查？

合规审查是**知识密集型**任务——需要检索法规、比对政策、评估风险。与 contract-approval-agent（路由型任务）形成互补，展示不同类型的 agent 能力。

### 为什么需要 RAG / agentic search？

合规判断需要参考外部知识（法规、政策、先例）。LLM 的训练数据不包含企业内部的合规政策，必须通过检索注入。这里的重点不是固定 top-k，而是让检索成为 agent 可调用的工具：证据不够就补搜，搜不到就降低置信度并转人工确认。

### 为什么多步推理？

合规审查不是单一判断，而是多步骤的证据闭环：识别关键条款 → 检索相关法规 → 判断证据是否充分 → 必要时补检索 → 综合评估 → 生成报告。

### 为什么 HITL？

高风险的合规审查（关联交易、大额担保）必须人工确认。合规审查的错误代价远高于审批路由——漏审一个条款可能导致法律风险。

### 为什么风险评分？

纯「通过/不通过」的二元判断不够。风险评分（低/中/高）让人工审查者可以优先处理高风险项，提高效率。

### 为什么独立报告生成？

合规审查的输出不是简单的「通过」，而是需要可审计的报告——审查了哪些条款、参考了哪些法规、发现了什么风险、给出什么建议。

### 为什么与 contract-approval-agent 分开？

两个项目展示不同的 agent 能力：
- contract-approval-agent：路由 + HITL（结构化决策）
- compliance-review-agent：RAG + 推理（知识密集决策）

分开比合并更能证明「不同场景需要不同架构」。

```
compliance-review-agent/
├── main.py                ← CLI 入口
├── src/
│   ├── state.py           ← Pydantic State
│   ├── graph.py           ← LangGraph 图组装（6 节点）
```
