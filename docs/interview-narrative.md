# 面试叙事文档

## 90 秒讲解

> 这是我基于多行业合规审查的通用模式搭的第二个 agent 项目。
>
> 第一个项目（合同审批路由）解决的是「走哪条路」——结构化决策，用规则+HITL。这个项目解决的是「是否合规」——知识密集型决策，需要检索法规、比对政策、评估风险。
>
> 核心链路是：材料解析 → 初始法规检索 → ReAct 判断证据是否充分 → 不足时按需补检索 → 风险评分 → 中高风险走人工确认 → 生成可审计的合规报告。
>
> 和第一个项目的区别：第一个项目的工具是查表（审批矩阵），这个项目的工具是检索（法规库）。第一个项目用规则优先，这个项目用 LLM 推理优先、规则兜底。
>
> 数据全部虚构，覆盖正常通过、关联方审查、高风险阻断、跨业务隔离四类分支。

## 设计决策话术

### Q: 为什么做合规审查而不是其他场景？

> 合规审查是知识密集型任务——需要检索法规、比对政策、综合判断。这和第一个项目（路由型任务）形成互补，展示不同类型的 agent 能力。面试官看到两个项目，会认为我理解「不同场景需要不同架构」。

### Q: RAG / agentic search 在这个项目里解决什么？

> LLM 的训练数据不包含企业内部的合规政策和行业法规，所以需要检索注入外部知识。但我这里不把它讲成固定 top-k RAG，而是讲成 agentic search：法规检索是一个工具，模型先看初始证据，觉得不够就换查询词补搜，最后把引用法规、风险因素和置信度写成结构化输出。

### Q: 怎么判断该用 grep / BM25 / 向量库？

> 第一性原理是语料有没有廉价、可迭代搜索的结构。代码天然可 grep，因为文件名、目录和符号命名就是索引；企业材料经常是 PDF、扫描件、会议纪要和口语化命名，关键词打不中，这时 embedding 是把文本变得可搜的低成本手段。我的判断不是反向量库，而是先给语料造结构：分类体系、metadata、版本、生效日期。向量库解决语料没结构的问题，分类体系是在源头上给语料造结构，两者互补。

### Q: 为什么法规库要预建，而不是每次 web search？

> 法规语料权威源少、变更低频、自带条文编号，最适合预建和版本化。合规结论必须能追溯到具体条文版本，实时 web search 来源不可控、结果不稳定，审计时复现不了。生产里我会做官方源监控、diff、版本化入库，并按查询时点有效过滤；法规变更后自动重跑受影响样例，把 freshness 放进 eval。

### Q: 为什么用 ReAct 而不是规则引擎？

> 合规审查的判断链很长：识别条款 → 检索法规 → 判断证据是否充分 → 必要时补检索 → 综合评估。这个过程需要 agent 能根据中间结果决定下一步做什么——比如发现「关联交易」后，主动去检索关联交易的专项法规；搜不到直接依据时，则降低置信度并转人工。规则引擎写不出这种动态证据闭环。

### Q: 风险评分怎么做的？

> 三级评分：低/中/高。低风险自动通过，中高风险走 HITL。评分依据是：检索到的法规要求 + 材料中的风险因素 + 推理过程中的发现。评分结果附带理由，可审计。

### Q: 和第一个项目的关系？

> 两个项目展示不同的 agent 能力：第一个是路由+HITL（结构化决策），第二个是 RAG+推理（知识密集决策）。分开比合并更能证明「我理解不同场景需要不同架构」。面试官看到两个项目，会认为我有架构判断力，不是只会一种模式。

## 反模式

- ❌「这个系统可以替代合规审查员」——不能，agent 是辅助工具
- ❌「LLM 可以判断是否合法」——不能，法律判断需要人
- ❌「RAG 就是把文档塞进去」——本项目口径是检索工具化、按需补搜、证据闭环
- ❌「agentic search 就是不需要向量库」——不对，向量库是弱结构语料的检索工具之一，地位变了，不是存在性变了
- ❌「法规每次实时外搜更 fresh」——不对，合规更需要权威源、版本化和可复现
- ❌「ReAct 比规则引擎好」——不是，是适用场景不同

## 落点

1. 你理解不同类型的任务需要不同的 agent 架构
2. 你知道 RAG 在知识密集场景中的位置
3. 你能做多步推理的设计（ReAct / Plan-and-Execute）
4. 你会评估风险并设计 HITL 策略
5. 你知道知识库 freshness 要进 eval，而不是停在「建库完成」
